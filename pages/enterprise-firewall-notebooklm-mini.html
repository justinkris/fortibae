NP vs CP vs SOC — What Each Chip Actually Does
After this: you'll never confuse NP7 (network processor) with CP9 (content processor) again.
NP7 CP9 SOC5 NP Direct Offload Conditions
Five minutes, one topic: FortiGate SPU chips. No model specs, just the architecture. NP (Network Processor) handles L3/L4 forwarding at line rate — it sees packet headers, not content. CP (Content Processor) handles computationally expensive content tasks: SSL handshake offload, IPS pattern matching, AV scanning. SOC chips combine both into a single die for lower-end models. Now the NP Direct architecture: explain why traffic between two ports on the same NP chip bypasses the CPU entirely, and why that same-NP-port requirement matters for high-throughput deployments. Walk through the exact conditions that prevent offload — what traffic types cannot be offloaded to NP and must go through the main CPU. Use CLI commands to identify which chip a FortiGate has. End with the one exam trap: SSL inspection always requires the CP processor, not NP, which is why enabling full inspection reduces NP throughput gains.

MINI · DOMAIN 2
The 5 FortiManager Install Operations
After this: you'll know which drawer each operation reads from and writes to — without hesitation.
Retrieve Import Install Quick Install Reinstall
Tight drill — five operations, five minutes. Use a filing-cabinet analogy throughout: FortiManager's internal database is one drawer, the device's running config is another, and the policy package is a third. Retrieve: pulls the running config from the device into the FortiManager database — the device is the source of truth, FortiManager is overwritten. Import: pulls policy and objects from the device's running config into a policy package — used once, when first adopting a device. Install: pushes the policy package from FortiManager down to the device — FortiManager is now the source of truth. Quick Install: pushes only device-level settings (no policy package) — useful for interface-level changes. Reinstall: reinstalls the last deployed policy package without changes — recovery operation. End with the one exam question trap: which operation does NOT touch the policy package at all?

Dynamic Objects and Metadata Variables
After this: you'll understand how one policy can deploy correctly to 200 branches with different IP schemes.
Dynamic Objects $variable Syntax Per-Device Mapping Normalized Interfaces
Five minutes on the FortiManager feature that makes scale possible. The problem: you have 200 branches, each with a different LAN subnet and a different WAN interface name. A single policy package can't hardcode those values. The solution: dynamic objects. Explain how a dynamic object is a placeholder in the policy — it looks like a normal address object, but its value is resolved per-device at install time using a per-device mapping. Then metadata variables: the dollar-sign variable syntax ($LAN_SUBNET, $WAN_INTERFACE) that appears in CLI templates and gets substituted with per-device values at push time. Explain normalised interfaces — how FortiManager maps a logical interface name like "WAN" to port2 on one device and port1 on another. Walk through the exact install-time resolution sequence. End with the one thing that breaks everything: forgetting to set the per-device mapping before pushing a template with unresolved variables.

The TLS Handshake, Step by Step
After this: you'll trace a TLS 1.2 handshake packet-by-packet and explain why TLS 1.3 is shorter.
Client Hello Server Hello Key Exchange TLS 1.2 vs 1.3 Cipher Suite Format
Five-minute drill: the TLS handshake as a back-and-forth conversation. Walk through TLS 1.2 step by step — Client Hello (I support these cipher suites and TLS versions), Server Hello (I pick this cipher suite), Certificate (here's my identity, verify me), Server Key Exchange (here's my DH public value), Client Key Exchange (here's mine, now we both derive the same session key), Finished (encrypted, proves the handshake wasn't tampered with). Contrast with TLS 1.3's 1-RTT handshake — the client guesses the key exchange algorithm upfront, so the handshake is one round trip shorter. Then decode a cipher suite string: TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384 — explain each segment (key exchange, authentication, bulk cipher, MAC). End with the FortiGate exam trap: TLS 1.0 and 1.1 are disabled by default in FortiOS 7.x — what does that break in legacy environments?

ISDB vs Application Control — Know the Difference
After this: you'll know exactly which inspection layer each uses and why ISDB appears in SD-WAN rules but App Control doesn't.
ISDB L3/L4 App Control L7 DPI Signatures SD-WAN Rules
Five-minute concept clarifier — this confusion costs exam marks. ISDB (Internet Service Database) is a Fortinet-maintained database of IP addresses and port ranges belonging to known services (Microsoft 365, AWS, Zoom). It operates at L3 and L4 only — no deep packet inspection, no SSL decryption required. Application Control uses DPI signatures to identify what an application is actually doing at L7 — it needs to see inside the packet, which means SSL inspection must be enabled for encrypted traffic. Use the analogy: ISDB is a bouncer with a list of approved addresses (no ID needed, just your ZIP code). App Control is a detective who watches what you do inside the venue. Now the key exam point: SD-WAN rules can use ISDB to steer traffic without requiring SSL inspection — that's why it's used for cloud app steering. App Control cannot be used directly in SD-WAN rules. Walk through one scenario where you'd use each and one where you'd use both together.

IPS Profile Construction

Filter vs Signature Target: Client/Server Severity Levels Action Options
Five minutes — build an IPS profile field by field. Start with the two ways to add signatures to a profile: by filter (select all signatures matching a category, OS, severity, or target) versus individual signature (explicitly named signatures by ID). Explain severity levels from critical down to informational and why you'd typically start blocking at high-and-above. Target field: client means the signature detects attacks against clients (browser exploits, drive-bys), server means attacks against servers (web app exploits, database attacks) — choosing the right target cuts false positives dramatically. Action per entry: monitor (log only), block (drop packet, log), reset (send TCP RST, log), quarantine (block source IP for a defined period). Walk through the right action for each severity tier. End with the hping3 validation test: what command would you run, what signature should fire, and what log entry proves IPS is working correctly?

OSPF Router Types — ABR vs ASBR
After this: you'll instantly identify what role each router plays just from its position in the network diagram.
Area 0 Backbone ABR ASBR Redistribution OFF
Five-minute drill: OSPF topology roles. Area 0 is the backbone every other OSPF area must connect through — it's the superhighway that all inter-area traffic crosses. An ABR (Area Border Router) sits with one interface in Area 0 and another in a non-backbone area — it's the on-ramp. It summarises LSAs between areas and can aggregate routes (route summarisation happens here). An ASBR (Autonomous System Boundary Router) imports external routes from outside OSPF — a BGP neighbour, a static route, a connected network — into the OSPF domain. One router can be both an ABR and ASBR simultaneously. Walk through what happens to a packet traversing from Area 2 to Area 3 — it must cross Area 0, so the path is: Area 2 router → ABR → Area 0 → ABR → Area 3 router. End with the exam trap: redistribution into OSPF is disabled by default in FortiOS — what's the CLI to enable it, and what cost does a redistributed external route receive?

BGP Path Selection and AS Path Manipulation
After this: you'll explain why the same prefix from two ISPs takes one path over the other — and how to change it.
Local Preference AS Path Length Origin MED AS Path Prepending
Five minutes — BGP path selection attributes in priority order. Local Preference (highest wins, influences outbound traffic, stays within an AS — like an internal preference score). AS Path Length (shortest wins — fewer hops through other autonomous systems is preferred). Origin (IGP beats EGP beats incomplete — how the route was originally injected). MED (lowest wins, sent to a neighbour to influence inbound traffic — like a suggested on-ramp priority). Walk through a concrete scenario: two EBGP paths to the same prefix, one via ISP-A (2-hop AS path) and one via ISP-B (3-hop AS path) — which wins and why? Then AS Path Prepending: explain how you artificially lengthen your own AS path when advertising to ISP-B to make ISP-A the preferred inbound path. End with the ECMP case in FortiOS: what must be true for BGP to install multiple paths and what CLI enables it?

MINI · DOMAIN 5
Dead Peer Detection — Three Modes, Three Scenarios
After this: you'll match on-demand, on-idle, and disabled to their correct use cases without second-guessing.
DPD On-Demand DPD On-Idle DPD Disabled Phase 1 Keepalive
Five-minute drill — DPD modes with zero fluff. DPD is the IPsec mechanism that detects whether the remote peer is still alive so dead tunnels are cleared. On-demand: FortiGate only sends DPD probes when it has traffic to send but hasn't heard from the peer — the probe is triggered by outbound traffic. Best for unpredictable, bursty traffic patterns where the tunnel may be silent for long periods. On-idle: probes are sent on a regular timer regardless of traffic — best for links with regular traffic where a silent failure would go undetected. Disabled: FortiGate never initiates DPD but will respond to the peer's probes — best for stable, directly connected point-to-point links where both ends are always reachable. Use the office analogy: on-demand is messaging a colleague only when you need something. On-idle is a scheduled daily check-in call. Disabled is leaving your door open and trusting they'll knock if they need you. End with the exam trap: which DPD mode risks leaving a stale Phase 2 SA in place longest after a peer failure?
Exam Weight: High

MINI · DOMAIN 5
ADVPN Shortcut Tunnel Negotiation Flow
After this: you'll trace every step from "Spoke A wants to reach Spoke B" to "shortcut tunnel is up."
Hair-Pin Traffic Shortcut Offer IKEv2 Direct auto-discovery-sender net-device enable
Five minutes — trace the ADVPN shortcut tunnel packet flow step by step with no detours. Step 1: Spoke A sends traffic to Spoke B's subnet. No shortcut exists yet, so the packet goes via the hub — hair-pinned through the hub's tunnel to Spoke B. Step 2: the hub detects cross-spoke traffic and sends a shortcut offer to both Spoke A and Spoke B. This is the auto-discovery-sender role — the hub is the matchmaker. Step 3: Spoke B receives the shortcut offer containing Spoke A's public IP. Step 4: Spoke A and Spoke B initiate a direct IKEv2 negotiation to each other — this requires net-device enable on the spoke so the FortiGate creates a virtual interface for the shortcut. Step 5: shortcut Phase 1 and Phase 2 complete. Step 6: subsequent traffic from Spoke A to Spoke B takes the shortcut directly, bypassing the hub. End with the exam trap: why does static routing on the hub break ADVPN, and what does dynamic routing actually need to learn?
Exam Weight: High
