LESSON 1 · INTRODUCTION TO NETWORK SECURITY ARCHITECTURE

Enterprise Firewall Solution & Fortinet Use Cases

The traditional perimeter-only approach is obsolete. Modern threats (zero-day, APT, polymorphic malware, insider threats) require a zero-trust strategy with deep inspection at every network layer. Fortinet's NGFW extends traditional L3/L4 ACLs with application-layer awareness and UTM profiles.

Core idea: Know which FortiGate series maps to which deployment role — this drives product selection questions on the exam.

FortiGate Product Families

The Five Use Cases

1
DCFW — Data Center Firewall — NGFW + segmentation (VRFs, VDOMs) + HW SSL inspection + hyperscale. Physical switches unified as one logical unit. Traffic: 10 Gbps–1 Tbps.
2
ISFW — Internal Segmentation Firewall — VLANs, VDOMs, micro-segmentation, integrated W/LAN (FortiAP). Campus environment. Focus on east-west traffic inspection.
3
Cloud Firewalls — FortiGate VM, FortiCNF, FortiSASE. IPsec + SSL + L2TP VPNs, Secure SD-WAN, multicloud (AWS, Azure, GCP, VMware, etc.). SaaS-delivered via FortiSASE.
4
Distributed NGFW — Multiple NGFWs across HQ + campus + branches. Adds 5G/LTE (FortiExtender), ZTNA (FortiGate + FortiClient + FortiAuthenticator), DDoS protection, Secure SD-WAN.
5
DEFW — Distributed Enterprise Firewall — Branch-only. Basic firewall + IPsec VPN back to HQ. Not designed for deep UTM. Use 100-series if branch needs more inspection.
FortiManager
Single, unified management console for all FortiGate devices across all deployment types (branch, campus, DC, cloud). Centrally manages policies, objects, firmware, and configuration.
!
COMMON MISTAKE
DEFW (30–90 series) is NOT designed for deep security analysis. If a branch requires IPS or SSL inspection, you need at least a 100-series ISFW model. Don't confuse DEFW capabilities with ISFW.
LESSON 2 · CENTRAL MANAGEMENT

FortiManager — Architecture, Dynamic Objects & Scripts

FortiManager centralizes policy management using a two-layer architecture: the device layer (per-device configuration) and the policy layer (policy packages shared across devices). Understanding which operations touch which layer is a core exam concept.

Core idea: Every FortiManager operation either reads from or writes to the device layer, the policy layer, or both. Know which is which before the exam.

ADOM Architecture

ADOM (Administrative Domain)
Virtual management instances on FortiManager. Without ADOMs enabled, everything lives in the default root ADOM. With ADOMs, each can run a different FortiOS firmware version (supporting same version and 2 versions back — e.g., 7.6 global supports 7.4 and 7.2).
  • Global Database — stores objects and policies applied across multiple ADOMs. All global objects use the reserved g- prefix.
  • Device layer — per-device configuration (interfaces, system settings). Provisioning templates, Fabric View, Extender Manager live here.
  • Policy layer — policy packages. VPN Manager, AP Manager, FortiSwitch Manager live here.
  • FortiGuard ADOM — single FortiGuard database serves all ADOMs regardless of version.

Key Operations — Device Layer vs Policy Layer

Dynamic Objects & Normalized Interfaces

Normalized interfaces allow the same policy layer interface name (e.g., LAN) to map to different physical ports on different FortiGate devices without rewriting policies.

  • Per-platform mapping — applies uniform settings to all devices of the same model type
  • Per-device mapping — customized for individual devices (e.g., LAN → port2 on FortiGate-A, LAN → port4 on FortiGate-B)
  • Dynamic object mapping — firewall address objects can have per-device IP ranges (e.g., LAN = 172.16.0.0/24 for Branch1, 192.168.0.0/24 for Branch2)
  • Metadata variables — use $variable_name syntax; shown with magnifying glass icon in GUI; supports per-device mapping
  • Not all objects support per-device mapping (e.g., web filter profiles do not)

Scripts — Four Use Cases

1
Remote FortiGate Directly (via CLI) — Script runs directly on FortiGate, then FortiManager auto-retrieves to sync the device database. Config Status shows Synchronized.
2
Device Database — Script modifies the device layer only; status changes to Modified. You must then install device settings to push to FortiGate.
3
Policy Package / ADOM Database — Script creates objects in the policy layer; flags the policy package as Modified. Install the policy package to push changes.
4
CLI Templates (Overwrite) — Overwrites both device and policy layer configurations. Metadata variables allow dynamic values per device. Marks both statuses as Modified.

ZTP and LTP

  • ZTP (Zero-Touch Provisioning) — FortiGate auto-connects to FortiManager over WAN/internet with no preconfiguration
  • LTP (Low-Touch Provisioning) — minimal pre-configuration needed before FortiGate can find FortiManager
  • Model device — placeholder created before real device arrives; matched by serial number or pre-shared key; auto-links when real device connects
  • Pre-run CLI templates — one-time only; auto-detach after quick install; ideal for initial bootstrap tasks
  • CLI templates — persistent; remain assigned after install; use for ongoing configurations
★
EXAM NOTE
Quick install bypasses the installation preview. Reinstall policy bypasses the wizard. Both are "fast path" options — remember which one applies to device settings vs. policy packages.
!
COMMON MISTAKE
Retrieve and Auto-retrieve do NOT update the policy package — they only capture changes into the device layer revision history. You must run Import afterward to move changes into the policy package.
LESSON 3 · VLANs AND VDOMs

Network Segmentation with VLANs and Virtual Domains

VLANs create logical broadcast domain separation over shared physical infrastructure. VDOMs partition a single FortiGate into multiple independent virtual firewalls, each with its own policies, routing, and security profiles.

Core idea: VLANs segment at L2; VDOMs segment at L3 (firewall level). Inter-VDOM links are virtual cables between VDOMs — no physical interface needed.

VLAN Standards at a Glance

TIP
FortiGate does NOT add VLAN tags on ingress — the upstream switch does. FortiGate only adds/modifies/removes VLAN tags on egress (after routing decision). L2 devices add/remove; L3 devices modify.

VDOM Types

1
Admin VDOM — management only; handles FortiGuard updates; NO Policy & Objects, Security Profiles, VPN, or User & Authentication modules. Use for management isolation (DCFW, ISFW).
2
Traffic VDOM — NAT mode — standard firewall operation; interfaces have IP addresses; most common VDOM type. Default root VDOM is the management VDOM (can be changed).
3
Traffic VDOM — Transparent mode — L2 insertion point; NO IP addresses on interfaces (only a management IP); uses forward-domain to subdivide broadcast domains when more than 2 VLANs are present.
4
LAN Extension VDOM — connects FortiGate/FortiExtender (connector) to a FortiGate access controller via VXLAN/IPsec. Only Network and VPN modules available.

Key VDOM Facts

  • Default: 10 VDOMs per FortiGate (additional VDOMs require license)
  • Enable multi-VDOM: config system global → set vdom-mode multi-vdom
  • Check VDOM limit: get system status | grep "Max number of virtual domains"
  • Enabling VDOMs creates VDOM Global and root VDOM — neither can be deleted
  • VDOM Global affects the entire device (interfaces, firmware, DNS); only top-level admins should modify it
  • Inter-VDOM link names auto-append 0 and 1 (e.g., VDOM_Link0 in root, VDOM_Link1 in VDOM1)
  • Global objects use the g- prefix — consistent UTM across all VDOMs; modifying one updates all
!
COMMON MISTAKE
802.1ad (double-tagging) is NOT the same as 802.3ad (LACP / link aggregation). The exam may present both — 802.1ad = Q-in-Q for VLANs; 802.3ad = bonding for bandwidth + redundancy.
LESSON 4 · HIGH AVAILABILITY

FGCP, FGSP, and VRRP — FortiGate HA Protocols

High availability prevents single points of failure. FortiGate supports three HA approaches: FGCP (proprietary Fortinet clustering), FGSP (session sync without a single cluster), and VRRP (industry-standard gateway redundancy). Knowing when to use each is the exam focus.

Core idea: FGCP is the go-to Fortinet HA. FGSP keeps sessions alive across independent FortiGates. VRRP provides default gateway redundancy. They solve different problems.

FGCP — FortiGate Clustering Protocol

  • Active-Passive (A-P) — one primary handles all traffic; secondary is on hot standby. Failover is fast but secondary is idle.
  • Active-Active (A-A) — primary distributes sessions to secondary via load balancing. Both devices process traffic.
  • All cluster members must share: same model, same firmware, same license level, same hard drive config, same operating mode (management VDOM)
  • HA group ID, group name, password, and heartbeat interfaces must match across all members
  • Best practice: use at least two heartbeat interfaces (different physical ports); initially convert DHCP/PPPoE interfaces to static

Primary Election (Override Disabled — Default)

Override DISABLED (default): 1. Most monitored ports up → wins 2. Higher HA uptime → wins (force failover: # diagnose sys ha reset-uptime) 3. Higher priority → wins 4. Higher serial number → wins

Primary Election (Override Enabled)

Override ENABLED (config system ha → set override enable): 1. Most monitored ports up → wins 2. Higher priority → wins ← priority checked BEFORE uptime 3. Higher HA uptime → wins 4. Higher serial number → wins Force failover: change the primary's HA priority value
★
EXAM NOTE
Override disabled = uptime matters more than priority. Override enabled = priority matters most (allows you to designate a "preferred primary" that always wins when healthy). Override is CLI-only — no GUI setting.
!
COMMON MISTAKE
If cluster members have different licenses, the cluster uses the lowest-level license. This can silently reduce capability — always match licenses across all cluster members.
LESSON 5 · DYNAMIC ROUTING PROTOCOLS

OSPF Features & BGP in Enterprise Networks

FortiGate supports full dynamic routing including OSPF (link-state, intra-AS) and BGP (path-vector, inter-AS). Both are used in enterprise data center and WAN deployments. The exam tests OSPF area design and BGP convergence optimization.

Core idea: OSPF = internal routing within areas (Dijkstra's SPF). BGP = routing between autonomous systems and across the internet. Both support redistribution, ECMP, and route filtering on FortiGate.

OSPF — Key Concepts

Link-State Protocol
Each OSPF router maintains an identical LSDB describing the full topology. Dijkstra's algorithm builds a shortest-path tree. Updates only sent on topology change (periodic refresh every 30 minutes otherwise).
Router TypeDefinitionKey Role
Backbone RouterHas at least one interface in Area 0Core routing
ABR (Area Border Router)Connects non-backbone areas to Area 0Route summarization via config range
ASBR (Autonomous System Boundary Router)Redistributes external routing protocols into OSPFControls external route injection
  • All OSPF networks must have at least one backbone area (Area 0)
  • External routes (E1/E2) not load-balanced by default — enable ECMP with set rfc1583-compatible enable
  • OSPF redistribution: config router ospf → config redistribute [connected|static|bgp|rip|isis] (all disabled by default)

BGP — Key Concepts

  • IBGP — BGP within the same Autonomous System (AS)
  • EBGP — BGP between different ASes (internet routing)
  • By default, FortiGate BGP does NOT advertise prefixes — must configure redistribution explicitly
  • ECMP with BGP — enable set ebgp-multipath enable or set ibgp-multipath enable for load balancing
  • Loopback as BGP source — loopback stays up even if a physical interface fails; use set update-source <loopback> + set ebgp-enforce-multihop enable (required since loopback is not the direct next-hop)
★
EXAM NOTE
OSPF ECMP for external routes requires rfc1583-compatible enable. BGP multipath requires ebgp-multipath enable. These are separate settings — don't mix them up on the exam.
→
TIP
Using a loopback as the BGP source interface requires multihop to be enabled because the loopback IP is not directly connected to the BGP peer — it's reachable only via the physical interface. FortiGate needs to know to look up the route to get there.
LESSON 6 · SECURITY PROFILES

SSL/SSH Inspection, Web Filter, Application Control, IPS & UTM

Security profiles apply deep packet inspection to network traffic. SSL/TLS inspection decrypts and re-encrypts HTTPS traffic so other profiles (IPS, web filter, antivirus) can analyze the plaintext payload.

Core idea: Without SSL inspection, FortiGate can't see inside encrypted traffic. Everything else (web filter, IPS, AV) is blind to HTTPS without it.

TLS Version Reference

VersionStatusPublishedNotes
SSL 2.0 / 3.0Deprecated1995–1996Never use
TLS 1.0 / 1.1Deprecated1999–2006Disabled on modern systems
TLS 1.2Current2008 (RFC 5246)Widely deployed, supported
TLS 1.3Preferred2018 (RFC 8446)Faster handshake, better cipher suites
  • Admin HTTPS TLS: config system global → set admin-https-ssl-versions tlsv1-2 tlsv1-3
  • TLS cipher suite format: TLS_[KeyExchange]_[Auth]_WITH_[Encryption]_[Hash]
  • HTTP/3 uses QUIC (UDP) instead of TCP; TLS 1.3 is mandatory (not optional)

Web Filter vs Application Control vs ISDB

FeatureOSI LayerInspectionUsed InFortiGuard Needed
Web FilterL7 (URLs / domains)Flow or ProxySecurity profile → internet policyYes (FortiGuard category ratings)
Application ControlL7 (app signatures)Flow-basedSecurity profile → internet policyYes (application definitions)
ISDBL3 + L4 (IP + port)No deep inspectionPolicy & Objects, SD-WAN rulesYes (internet service DB)
ISDB (Internet Service Database)
A database of known IP ranges and port patterns for major internet services (Google, Microsoft, Zoom, etc.). Works at L3/L4 only — no payload inspection. Used in policies and SD-WAN rules when you want to steer or block traffic by service identity rather than URL.

IPS — Intrusion Prevention System

  • IPS profiles use filters (by severity, protocol, OS, target) or signatures (specific CVEs)
  • Targets: Server (protect servers from client attacks) and/or Client (protect clients from server-delivered exploits)
  • Enabling both targets + all severities + Block action = maximum protection but highest false-positive risk
  • Validate IPS with hping3 — open-source ethical attack simulator for crafting custom TCP/IP packets
!
COMMON MISTAKE
Web filter and application control both require FortiGuard licenses and depend on FortiGuard lookups. ISDB also requires FortiGuard. Without an active FortiGuard subscription, all three features degrade or stop working.
LESSON 7 · IPsec

IPsec Topology Design & FortiManager IPsec Templates

IPsec provides encrypted, authenticated tunnels between FortiGate devices. Topology selection (site-to-site, hub-and-spoke, full mesh, ADVPN) determines scalability, latency, and management complexity.

Core idea: Choose IKEv2 for all new deployments — it's faster, more secure, and supports EAP authentication. Hub-and-spoke scales easily but introduces hub bottlenecks; ADVPN solves this by dynamically creating spoke-to-spoke tunnels.

IPsec Topology Options

TopologyBest ForTunnels NeededScalability
Site-to-SiteTwo fixed locations1Low
Hub-and-SpokeMany branches to one HQN (one per spoke)Good — simple management
Dual Hub-and-SpokeHub redundancy2NGood — resilient
Full MeshLow-latency any-to-anyN(N-1)/2Poor — complex at scale
ADVPNDynamic spoke-to-spokeN + dynamic shortcutsExcellent — best of both

IKEv1 vs IKEv2

FeatureIKEv1IKEv2
EncryptionDES, 3DES, AES-CBCAES-GCM, ChaCha20P
AuthenticationPSK, Digital SignaturesEAP, PSK, Digital Signatures
IntegrityHMAC-MD5, HMAC-SHA1HMAC-SHA2, AES-XCBC
DH GroupsGroups 1, 2, 5 (weak)Groups 14–20 (strong, includes ECP)

DH Groups — MODP vs ECP

  • MODP (Modular Exponential) — Groups 1, 2, 5, 14–18. Strength increases with key length. Group 14 (2048-bit) is the minimum recommended.
  • ECP (Elliptic Curve) — Groups 19 (256-bit), 20 (384-bit). Shorter keys, equivalent or better security. Preferred for modern deployments.
  • PFS (Perfect Forward Secrecy) — enabled in Phase 2 with a DH group. Ensures compromise of one session does not compromise past sessions.

DPD — Dead Peer Detection

DPD ModeWhen to UseBehavior
On-demandUnpredictable/bursty trafficSends DPD only when traffic needs to flow but no response received
On-idleRegular, periodic traffic patternsProactively sends DPD during idle periods to verify peer
DisabledStable networks with alternate monitoringOnly replies to DPDs initiated by the peer; never initiates
★
EXAM NOTE
DH Groups 1, 2, and 5 are considered weak and should never be used in new deployments. The exam may present a scenario asking you to identify the most secure configuration — always prefer ECP (Group 19/20) or MODP Group 14+ for IKEv2.
LESSON 8 · AUTO-DISCOVERY VPN

ADVPN — Dynamic Spoke-to-Spoke Tunnels

ADVPN combines the management simplicity of hub-and-spoke with the performance of full-mesh. The hub acts as a broker: when Spoke A needs to reach Spoke B, the hub redirects them to negotiate a direct IPsec shortcut tunnel, eliminating hub-transit latency.

Core idea: ADVPN dynamically creates on-demand IPsec shortcuts between spokes. The hub still manages connectivity but traffic flows directly between spokes once the shortcut is established.

ADVPN vs Traditional Topologies

Hub-Spoke
Simple Management
All spoke traffic transits hub. Hub becomes bottleneck as spokes scale.
Full Mesh
Best Performance
Direct paths everywhere but N(N-1)/2 tunnels — unmanageable at scale.
ADVPN
Best of Both
Hub-based setup + automatic spoke-to-spoke shortcuts on demand.

  • Dynamic routing (BGP or OSPF over the VPN) is used on the hub to learn spoke subnets — static routes are not scalable
  • Spoke-to-spoke shortcut tunnels are created automatically when the hub redirects spoke traffic
  • FortiManager IPsec templates simplify ADVPN setup — activate hub or branch template, enable ADVPN option, configure metadata variables for tunnel IPs

TIP
ADVPN phase 1 on the hub uses type dynamic so it can accept connections from any spoke without pre-configuring each one. Spokes connect to the hub's static IP but then shortcut directly to each other.
LESSON 9 · SECURITY FABRIC

Fabric Connectors, External Connectors & Automation Stitches

The Fortinet Security Fabric is a cybersecurity mesh platform built on FortiOS. It integrates Fortinet products and third-party solutions to share intelligence, automate responses, and enforce consistent policy across the entire attack surface.

Core idea: Security Fabric = Broad (visibility everywhere) + Integrated (all products communicate) + Automated (faster response without human intervention). Fabric Connectors are for Fortinet-native integration; External Connectors are for third-party cloud/SDN/endpoint systems.

Fabric Connectors vs External Connectors

FeatureFabric ConnectorsExternal Connectors
Location in GUISecurity Fabric → Fabric ConnectorsSecurity Fabric → External Connectors
ExamplesFortiGate, FortiAnalyzer, FortiClient EMS, FortiManager, FortiSandboxAWS, GCP, Azure, Kubernetes, VMware, FSSO, RADIUS, threat feeds
Key ActionsAuto-quarantine, automated remediation across Fortinet productsAuto-update policies based on cloud environment changes (SDN)
Threat Feeds—FortiGuard category, Malware Hash, IP/Domain/URL feeds

External Feeds

  • Import plain-text lists via HTTP(S) or STIX format
  • FortiOS periodically fetches updates from the URI and applies changes immediately
  • Feed types: FortiGuard category, IP addresses, domain names, MAC addresses, malware hashes
  • External feed creates a resource object usable directly in firewall policies

Automation Stitches

  • Stitches combine a trigger (event) + one or more actions (response)
  • Example trigger: IOC (Indicator of Compromise) detected by FortiAnalyzer
  • Example actions: quarantine host, block IP, send email alert, run a script
  • Enables faster time-to-prevention without manual administrator intervention

LESSON 10 · HARDWARE ACCELERATION ON FORTIGATE

SPU Architecture, Traffic Offload & Configuring HW Acceleration

Fortinet's Security Processing Units (SPU) offload computationally intensive tasks (encryption, decryption, packet inspection) from the main CPU to dedicated hardware processors. This delivers line-rate performance even with deep inspection enabled.

Core idea: NP processors handle network/forwarding tasks; CP processors handle security/content tasks. Traffic offloaded to NP/CP bypasses the main CPU, enabling much higher throughput. Not all traffic can be offloaded.

SPU Types

ProcessorTypePrimary FunctionExamples
NP6Network ProcessorPacket forwarding, session offload, IPsecFortiGate 200E, 2000E, 2500E
NP7Network Processor (next-gen)Same as NP6 + higher throughput, more featuresFortiGate 400F, 600F, 900G
CP9Content ProcessorSSL/TLS inspection, IPS, antivirus accelerationFortiGate 900G (ASIC version: CP9)
SP5 / SOC5Security Processor (SoC)Entry/mid-range — combines NP + CP on one chipFortiGate 91G (ASIC version: SOC5)

Identifying SPU Type via CLI

# Check NP model (vendor ID 1a29 = Fortinet): # diagnose hardware lspci | grep 1a29 → If NPID is 4e36 = NP6 → If NPID is 4e37 = NP7 # Check CP model: # get hardware status → Look for "ASIC version: CP9" (Content Processor version) # Confirm SP5 (SoC): # get hardware status → Look for "ASIC version: SOC5"

NP Direct Architecture

  • Hardware design where NP processor connects directly to physical interfaces — eliminates the internal switch fabric (ISF) hop
  • Reduces forwarding latency to ultra-low levels
  • NP6 direct: available on FortiGate 200E, 2000E, 2500E (two or more NP6s)
  • NP7 direct: available on FortiGate 400F, 600F, 900G (ports X5–X8 directly wired to NP7)
  • Physical topology requirement: all traffic must enter and exit through interfaces on the same NP processor to be offloaded
→
TIP
If traffic enters on a port attached to NP7 but exits on a port attached to a different NP — it cannot be offloaded. Design your physical cabling so that all traffic for a given flow uses interfaces on the same NP chip.
★
EXAM NOTE
NP = network forwarding offload. CP = security inspection offload. SP/SOC = combined chip for smaller models. The exam may ask you to identify the correct processor type for a given capability (e.g., "which processor accelerates SSL inspection?" → CP).
EXAM TIPS

What the Exam Focuses On

The Enterprise Firewall 7.6 exam is scenario-based. You will be given a customer requirement and must identify the correct Fortinet product, mode, or configuration. Memorizing the "why" behind each feature is more valuable than memorizing CLI syntax.

★
TOP EXAM SCENARIOS TO KNOW
1. "Branch needs basic firewall + VPN" → DEFW (30–90 series)
2. "Campus needs micro-segmentation + wireless" → ISFW (100–900 series)
3. "Customer wants same policy across 50 branches" → FortiManager + policy packages + normalized interfaces
4. "Need UTM without changing IP addresses or routes" → Transparent mode VDOM
5. "Hub-and-spoke VPN but spokes need to talk to each other directly" → ADVPN
6. "FortiGate cluster: always want Device A as primary" → Override enabled + highest priority on Device A

Quick-Fire Exam Distinctions

  • 802.1ad ≠ 802.3ad — 802.1ad = Q-in-Q VLAN tagging; 802.3ad = LACP link aggregation
  • Retrieve ≠ Import — Retrieve captures to device layer revision only; Import moves to policy package
  • Pre-run CLI template ≠ CLI template — pre-run auto-detaches after install; CLI template stays assigned
  • Override disabled = uptime wins; Override enabled = priority wins
  • Web filter = URLs/domains; App control = app signatures; ISDB = IP+port only (no deep inspection)
  • NP = forwarding; CP = inspection; SOC = combined entry-level
  • Admin VDOM = no Policy & Objects, no VPN, no User Auth
  • ADVPN requires IKEv2 — hub uses type dynamic; spokes use auto-discovery-receiver enable
  • OSPF ECMP external routes — need set rfc1583-compatible enable
  • BGP ECMP — need set ebgp-multipath enable or set ibgp-multipath enable
  • Loopback as BGP source — requires ebgp-enforce-multihop enable
  • Global objects prefix — g- reserved on both FortiGate and FortiManager
SCENARIO — EXAM QUESTION STYLE
A company has 200 branch offices. Branch admins sometimes make changes directly on local FortiGates. The central security team needs to capture those local changes into FortiManager without disrupting the policy package. What is the correct operation?
Answer: Run Retrieve (or enable Auto-retrieve). This captures the FortiGate's current running config into the device layer revision history without modifying the policy package. The team can then optionally run Import afterward if they want to move changes into the policy package.
SCENARIO — EXAM QUESTION STYLE
A customer needs to inspect encrypted web traffic on a transparent firewall inserted between a switch and a router without changing any IP addresses. Which FortiGate feature meets this requirement?
Answer: A Traffic VDOM in Transparent mode. It operates at L2 so no IP addresses are assigned to interfaces, the network topology is unchanged, and FortiGate can still apply SSL inspection and security profiles to passing traffic.

Transparent VDOM = no IP on interfaces (management IP only) + forward-domain for VLAN separation + L2 insertion without topology changes. NAT VDOM = interfaces have IPs + acts as a full L3 firewall.
